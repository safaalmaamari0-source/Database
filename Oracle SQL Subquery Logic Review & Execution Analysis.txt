------Final Reflection------

--Why subqueries are useful in enterprise HR reporting systems.
Subqueries allow HR reports to dynamically calculate thresholds like averages, maximums, and counts at runtime,
so reports always reflect current data without hardcoded values.

--The difference between JOIN-based thinking and subquery-based thinking.
JOIN thinking combines tables side by side to produce wider rows, while subquery thinking answers an inner question first and uses that result to filter the outer question.

--Why some subqueries produce runtime errors.
Runtime errors occur when there is a mismatch between the operator used and what the subquery actually returns — 
such as using = when the subquery returns multiple rows, or placing a group function like AVG() directly in WHERE before grouping has occurred.

--Why SQL execution order is important.
SQL does not execute in the order it is written — it follows a fixed internal sequence of FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY — 
and understanding this order explains why certain functions and aliases are available in some clauses but not others.

--Why understanding query logic is more valuable than memorizing syntax.
Syntax can be looked up in seconds and varies between databases, but query logic — knowing what question to ask, which operator fits the result, 
and whether the business requirement is correctly expressed — is the skill that determines whether your query returns the right answer.
