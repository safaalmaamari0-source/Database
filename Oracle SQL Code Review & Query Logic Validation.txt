FROM employees e

ON e.department_id = d.department_id;

Query Review 3:
--Analyze the returned results--
This query has a silent but critical logical contradiction, the WHERE clause kills the LEFT JOIN. The WHERE clause filters
 on a column from the right table. Since NULL ≠ 'Sales', all unassigned employees are silently dropped — the LEFT JOIN is 
effectively converted into an INNER JOIN.

--Determine whether the business logic is correct--
 The Business Logic Is Contradictory, the query contains two conflicting business intentions written in the same statement,
these two intentions directly contradict each other and cannot both be satisfied simultaneously.

--Rewrite or improve the query if necessary--
SELECT e.first_name, d.department_name
FROM employees e
INNER JOIN departments d
    ON e.department_id = d.department_id
WHERE d.department_name = 'Sales';

--Briefly explain why the original query was incorrect and why the revised version is better--
* The original query used LEFT JOIN to keep all employees but then used WHERE d.department_name = 'Sales' 
which filtered out all NULL rows — silently defeating the purpose of the LEFT JOIN and behaving exactly like an INNER JOIN anyway.
 The fix either uses INNER JOIN explicitly when only Sales employees are needed, or moves the filter into the ON clause when all 
employees need to be preserved.


Query Review 4:

--Analyze the returned results--
The query attempts a self join on the employees table, which is the correct approach for finding manager relationships. 
However the ON clause is reversed, this is backwards. It finds employees whose ID matches someone else's manager_id — 
meaning it returns managers and who reports to them, not employees and who manages them.

--Determine whether the business logic is correct--
The business logic is broken because the JOIN condition is written in the wrong direction. 
The query is supposed to answer the simple business question.

--Rewrite or improve the query if necessary--
SELECT 
    e.first_name  AS employee_name,
    m.first_name  AS manager_name
FROM employees e
JOIN employees m
ON e.manager_id = m.employee_id;

--Briefly explain why the original query was incorrect and why the revised version is better--
The original query had the JOIN condition written backwards — e.employee_id = m.manager_id — 
which flipped the entire relationship, mistakenly showing managers as employees and employees as managers. 
The fix corrects this by reversing the condition to e.manager_id = m.employee_id, which properly follows the natural 
foreign key to primary key direction, ensuring each employee is correctly paired with their actual manager. 
Adding a LEFT JOIN with COALESCE further improves the query by making sure the top-level employee like the CEO, 
who has no manager, is never silently dropped from the results.
